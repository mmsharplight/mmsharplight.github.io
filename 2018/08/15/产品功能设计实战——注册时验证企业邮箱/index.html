产品功能设计实战——注册时验证企业邮箱

发表于 2018-08-15 |

  我们通常在对产品用户群进行分析时，总是希望获取到尽可能多的用户信息，有利于我们对用户进行分层，进行更加精准的用户画像的绘制。注册时验证或者绑定企业邮箱可以知道用户的行业，可能平时更关注哪些内容等，进行更加精准的运营。

  本文主要分享如果要在注册时验证企业邮箱，如何功能的分析和设计。

1、确定功能的目的

  首先目的可以分为两个层面，一个站在用户角度，一个站在产品角度。

  对用户的意义

  • 可以增加一个登录方式
  • 可以获得产品的更新迭代信息
  • 可以找到同企业都在使用同一产品的人

  对产品本身的意义

  • 可以做更加精准的电子邮箱营销（EDM）
    • 个性化：EDM是一对一的沟通，让用户感觉到尊重，感觉到这是为他所建立并且是他所独享的沟通方式
    • 用户关注：用户关注产品的哪些方面，愿意掏钱的点在哪
    • 用户嗜好：针对用户喜爱的内容（兴趣方面），与用户建立长久的良性的互动关系
  • 召回流失用户（用户留存）

2、明确功能基本逻辑

  • 用户逻辑：用户使用业务逻辑
    1. 填写邮箱
    2. 区分企业邮箱
    3. 下发验证信息
    4. 用户确认
    5. 验证成功
  • 数据逻辑:后台数据流转
    1. 填写邮箱
    2. 验证企业邮箱：验证方法如下
    3. 生成验证信息：要看发送验证渠道是否通畅，不然用户体验很差
    4. 验证成功后用户身份状态信息要更改

  验证企业邮箱有两个方法：

  1. 命中法：收集大量企业邮箱，组成企业邮箱库
  2. 排除法：排除163、126等主流邮箱

3、设计关键模块方案

3.1 下发验证功能流程

  下发验证的功能可以通过下面三个方案进行实现。

  • 方案一：生成验证链接，链接通过邮件发送，用户点击链接进入，即可验证成功
    • 优点
      • 用户操作起来简单方便
    • 缺点
      • 链接可能会被当成广告，会被拦截
  • 方案二：生成验证码，验证码发送至验证邮箱，用户将验证码填写后提交，即可验证成功
    • 优点
      • 邮件被拦截的概率大大降低
    • 缺点
      • 用户操作起来复杂
  • 方案三：生成验证码后，用户通过验证的邮箱主动发送验证码，匹配成果后，则验证通过
    • 缺点
      • 用户操作起来复杂
      • 产品进行验证码的反解析复杂

3.2 常用方案细化

  上述的三个方案中，相比之下方案二较好。因此对方案二进行流程细化。

小延伸：除了像神策数据、GrowingIo这类的企业级产品，大多数产品在用户注册时，不会要求验证企业邮箱的，如何在使用过程中去引导用户去绑定企业邮箱呢？此处就需要策略，策略方面我涉及还不是很多，最常见的以下两种：

  • 利益层面引导：这个是最常见的，绑定后会给予积分奖励或者优惠券补贴等，很简单，但往往最有效。
  • 心理层面引导：告知用户绑定企业邮箱的好处，比如可以得知同公司有哪些人在与你使用相同的产品。但在这个隐私至上的时代，很多人会不愿意透露太多信息。
